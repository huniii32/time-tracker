# 작업명

Milestone 10 설정 기능 MVP 추가

# 작업 목표

`Time Tracker` 앱에 사용자별 설정 화면을 추가하고, 입사일, 사용자 정보, 목표 기록 시간, 주간 기준 요일, 기본 시작 화면을 현재 로그인한 사용자 기준으로 조회/저장할 수 있게 한다.

# 구현한 것

- `/settings` 설정 페이지 추가
- 설정 화면 상단 히어로와 카드형 섹션 UI 추가
- 프로필 설정 입력: 이름, 입사일, 직무, 소속 팀
- 시간 기준 설정 입력: 하루 목표 기록 시간, 주간 목표 기록 시간, 주간 시작 요일
- 앱 표시 설정 입력: 기본 시작 화면
- 설정 row가 없을 때 기본값으로 폼 초기화
- 현재 로그인 사용자 기준 `user_settings` 조회
- 현재 로그인 사용자 기준 `user_settings` upsert 저장
- 비로그인 사용자의 `/settings` 접근 시 `/login` 이동
- Header 우측 로그아웃 영역 근처에 설정 링크 추가
- 홈 대시보드에 설정값 반영
- 설정 입사일을 홈 대시보드 D+일차 계산에 우선 반영
- 설정 이름을 홈 환영 문구에 반영
- 하루 목표 기록 시간을 오늘 기록 시간 카드에 표시
- 주간 목표 기록 시간을 이번 주 시간 요약 카드에 표시
- 주간 시작 요일 기준으로 이번 주 시간 기록 범위 계산
- `src/app`, `src/components`에서 직접 Supabase `.from(...)` 호출 없이 query layer 사용

# DB/RLS 변경 필요 여부

필요함.

실제 Supabase DB에 `user_settings` 테이블과 사용자별 RLS 정책을 적용해야 `/settings` 조회/저장이 동작한다.

필요 스키마 방향:

```sql
create table if not exists user_settings (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  display_name text,
  joined_at date,
  role text,
  team text,
  daily_target_hours numeric default 8,
  weekly_target_hours numeric default 40,
  week_starts_on text default 'monday',
  default_start_page text default 'home',
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  unique(user_id)
);
```

필요 RLS 방향:

```sql
alter table user_settings enable row level security;

create policy "Users can view own settings"
on user_settings for select
using (auth.uid() = user_id);

create policy "Users can insert own settings"
on user_settings for insert
with check (auth.uid() = user_id);

create policy "Users can update own settings"
on user_settings for update
using (auth.uid() = user_id)
with check (auth.uid() = user_id);

create policy "Users can delete own settings"
on user_settings for delete
using (auth.uid() = user_id);
```

# 검증 결과

- 타입 체크 통과
- 린트 통과
- 빌드 통과
- 첫 빌드는 PowerShell 샌드박스에서 `spawn EPERM`으로 실패했고, 승인된 외부 실행에서 통과
- 빌드 결과에서 `/settings` 라우트 포함 확인
- `src/app`, `src/components` 내 직접 `.from(...)` 호출 없음
- 설정 조회/저장은 `src/lib/queries/settings.ts` query layer 사용
- 설정 조회/저장과 홈 대시보드 설정 조회에서 현재 로그인 사용자 `user.id` 사용 확인
- 기존 CRUD/auth 호출은 기존 컴포넌트의 기존 query/auth 흐름만 검색됨
- `package.json` 의존성 변경 없음

# 남은 이슈

- 실제 Supabase DB에 `user_settings` 테이블과 RLS 정책을 적용해야 런타임 저장/조회가 가능하다.
- 실제 로그인 계정으로 `/settings` 저장, 새로고침 유지, 계정 간 격리 수동 검증은 수행하지 않았다.
- 로그인 후 기본 시작 화면 자동 이동은 이번 범위에서 제외했다.
