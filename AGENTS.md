# AGENTS.md

## 프로젝트 목적

Time Tracker는 개인용 회사 적응 및 시간 기록 웹앱이다.

사용자가 로그인한 뒤 업무, 노트, 루틴, 회고, 목표, 미팅 기록을 단계적으로 관리하고, 자신의 시간 사용 패턴과 회사 적응 과정을 구조적으로 기록할 수 있게 만든다.

이 프로젝트는 단순 일정 관리 앱이 아니라, 아래 흐름을 중심으로 한다.

입력 → 기록 → 시간화 → 회고 → 개선

## 제품 방향

Time Tracker는 아래 목적을 가진다.

- 회사 적응 과정에서 발생하는 업무, 사람, 용어, 학습 내용을 기록한다.
- 하루 단위의 시간 사용을 기록하고 누적한다.
- 루틴과 목표를 관리해 반복 행동을 추적한다.
- 회고를 통해 배운 점, 어려웠던 점, 감정 관리 포인트를 정리한다.
- 기록 데이터를 기반으로 사용자가 다음 행동을 개선할 수 있게 한다.

## 작업 원칙

- 전체 MVP를 한 번에 구현하지 않는다.
- 한 번의 작업은 하나의 마일스톤 또는 명시된 작업 범위만 진행한다.
- 요청하지 않은 기능을 구현하지 않는다.
- 현재 마일스톤에 포함되지 않은 기능은 구현하지 않는다.
- PRD는 제품 전체 방향을 설명하는 참고 문서로만 사용한다.
- 실제 구현 범위는 `docs/MILESTONES.md`와 사용자가 요청한 작업 범위를 우선한다.
- 구현 후에는 반드시 `docs/HARNESS.md` 기준으로 검증한다.
- 검증 실패 시 완료 보고하지 않는다.
- 기존 기능이 깨지는 변경은 하지 않는다.
- 기존 라우트 구조, 인증 흐름, query layer 구조를 임의로 변경하지 않는다.
- DB schema 변경, RLS policy 변경, 인증 구조 변경은 명시 요청이 있을 때만 수행한다.

## 마일스톤 해석 원칙

`docs/MILESTONES.md`는 현재 구현 범위를 정의하는 기준 문서다.

- 현재 마일스톤에 포함된 기능만 구현한다.
- 제외 범위에 있는 기능은 구현하지 않는다.
- 다음 마일스톤의 기능을 미리 구현하지 않는다.
- 관련 기능처럼 보여도 현재 범위에 없으면 구현하지 않는다.
- 모호한 경우에는 범위를 넓히지 말고 최소 구현으로 제한한다.

## PRD 해석 원칙

`docs/PRD.md`는 제품 전체 방향을 설명한다.

PRD에 있는 기능이라도 현재 마일스톤에 포함되지 않으면 구현하지 않는다.

예를 들어 PRD에 AI 요약, 차트, 캘린더 연동, 알림, 앱 출시 관련 내용이 있더라도 현재 마일스톤에 없다면 구현하지 않는다.

## Supabase 작업 원칙

Supabase 관련 작업은 반드시 기존 query layer 구조를 따른다.

- 컴포넌트에서 직접 `.from(...)`을 호출하지 않는다.
- `src/app`, `src/components` 내부에서 Supabase table query를 직접 실행하지 않는다.
- 데이터 조회, 생성, 수정, 삭제는 기존 query layer 또는 domain query 함수를 사용한다.
- 현재 로그인한 사용자 기준으로 `user.id`를 전달한다.
- 사용자별 데이터 접근은 `user_id` 기준으로 제한한다.
- 인증 흐름은 기존 구조를 유지한다.
- RLS policy는 명시 요청 없이는 변경하지 않는다.
- DB schema는 명시 요청 없이는 변경하지 않는다.

## 컴포넌트 작업 원칙

- 기존 UI 디자인 방향을 유지한다.
- Time Tracker의 네이비/화이트 톤을 유지한다.
- 대규모 UI 리디자인은 명시 요청이 있을 때만 진행한다.
- 기존 컴포넌트를 삭제하지 않는다.
- 재사용 가능한 UI는 컴포넌트로 분리한다.
- 페이지 컴포넌트는 가능한 한 데이터 흐름과 레이아웃 중심으로 유지한다.
- 비즈니스 로직은 컴포넌트 안에 과도하게 넣지 않는다.
- 타입 안정성을 유지한다.
- 모바일 화면에서 깨지지 않도록 반응형을 고려한다.

## CRUD 작업 원칙

CRUD 기능을 구현할 때는 아래 흐름을 유지한다.

- 목록 조회
- 상세 조회
- 생성
- 수정
- 삭제
- 빈 상태 처리
- 로딩 상태 처리
- 에러 상태 처리
- 로그인 사용자 기준 데이터 필터링

단, 현재 마일스톤에서 요청된 CRUD만 구현한다.

다른 도메인의 CRUD를 함께 추가하거나 수정하지 않는다.

## 이미지 및 파일 첨부 작업 원칙

이미지 첨부 기능을 구현할 때는 현재 마일스톤 범위를 따른다.

- 백엔드 저장이 명시되지 않은 경우 프론트엔드 미리보기 중심으로 구현한다.
- 실제 파일 저장소 연동은 명시 요청이 있을 때만 구현한다.
- Supabase Storage 연동은 별도 마일스톤으로 분리한다.
- 이미지 관련 로직은 추후 서버 연동이 가능하도록 함수로 분리한다.
- 이미지 미리보기, 삭제, 설명 입력은 UI 범위에서 구현할 수 있다.
- DB schema 변경 없이 가능한 구조를 우선한다.

## 캘린더 작업 원칙

캘린더 기능은 내부 표시용 캘린더와 외부 캘린더 연동을 구분한다.

- 월간/주간 UI 표시용 캘린더는 현재 범위에 포함될 수 있다.
- Google Calendar, Apple Calendar 등 외부 연동은 명시 요청 없이는 구현하지 않는다.
- 캘린더 데이터는 기존 루틴, 시간 기록, 목표 데이터 기반으로 표시한다.
- 외부 API 연동이나 OAuth 추가는 하지 않는다.

## 검증 원칙

작업 후 반드시 `docs/HARNESS.md` 기준으로 검증한다.

가능하면 다음 항목을 확인한다.

- 타입 체크
- 린트
- Codex 환경에서는 빌드를 실행하지 않는다.
- 빌드 검증은 사용자가 로컬 환경에서 직접 수행한다.
- 주요 라우트 렌더링
- 현재 작업 대상 기능 동작
- 기존 CRUD 기능 유지 여부
- 로그인 사용자 기준 데이터 조회 여부
- query layer 사용 여부
- `src/app`, `src/components` 내 직접 `.from(...)` 호출 여부
- 범위 밖 기능 구현 여부
- 불필요한 라이브러리 추가 여부

검증 실패 시 완료 보고하지 않는다.

검증 실패가 발생하면 실패 항목, 원인, 남은 조치를 작업 로그에 기록한다.

## 작업 로그 규칙

모든 작업 완료 후 반드시 `logs/` 폴더에 작업 로그를 남긴다.

파일명은 다음 형식을 따른다.

`logs/YYYY-MM-DD-HHMM-[work-name].md`

예시:

`logs/2026-05-05-2130-milestone-1-auth-foundation.md`

`logs/2026-05-09-1840-note-routine-structure-improvement.md`

작업 로그에는 다음 항목을 포함한다.

# 작업 로그: [작업명]

## 작업 목표

## 구현한 것

## 구현하지 않은 것

## 생성한 파일

## 수정한 파일

## 실행한 검증

## 검증 결과

## 범위 초과 여부

## 현재 한계

## 다음 추천 작업

## 채팅 보고 규칙

작업 완료 후 채팅에는 아래 항목만 짧게 보고한다.

- 완료 여부
- 생성한 로그 파일 경로
- 검증 결과 요약
- 현재 한계
- 다음 추천 작업

상세한 구현 내용은 채팅에 길게 작성하지 않고 작업 로그에 기록한다.

## 완료 보고 기준

다음 조건을 모두 만족해야 완료로 보고할 수 있다.

- 요청된 범위가 구현되었다.
- 범위 밖 기능을 구현하지 않았다.
- 기존 기능이 깨지지 않았다.
- 타입 체크가 통과했다.
- 린트가 통과했다.
- Codex 환경에서 요구되는 검증이 통과했다.
- 빌드 검증은 사용자가 로컬 환경에서 직접 수행한다.
- `docs/HARNESS.md` 기준 검증을 수행했다.
- 작업 로그를 생성했다.

하나라도 충족하지 못하면 완료가 아니라 부분 완료 또는 검증 실패로 보고한다.

## 현재 프로젝트 운영 원칙

이 프로젝트는 빠르게 기능을 많이 넣는 것보다, 작게 나누어 안정적으로 확장하는 것을 우선한다.

따라서 모든 작업은 아래 기준을 따른다.

작은 범위
→ 명확한 구현
→ 기존 구조 유지
→ 검증
→ 로그 기록
→ 다음 마일스톤 진행

## 추후 확장 기능 처리 원칙

FastAPI 백엔드, AI 기능, Supabase Storage, 모바일 앱, 외부 캘린더 연동, 푸시 알림, 데이터 분석 리포트 등은 추후 확장 후보로 볼 수 있다.

단, 해당 기능들은 현재 마일스톤에 명시되어 있지 않으면 구현하지 않는다.

추후 확장 후보는 제품 방향 검토용이며, 실제 구현은 반드시 `docs/MILESTONES.md` 또는 별도 작업 요청에 포함되어야 한다.

PRD나 ROADMAP에 언급되어 있더라도 현재 마일스톤 범위에 없으면 구현하지 않는다.
