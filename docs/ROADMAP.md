# ROADMAP: Time Tracker

> 이 문서는 Time Tracker의 중장기 제품 확장 방향을 정리한 로드맵 문서다.
> 실제 구현 범위는 항상 `docs/MILESTONES.md`와 사용자의 명시 요청을 따른다.
> 이 문서에 포함된 기능이라도 현재 마일스톤에 포함되지 않으면 구현하지 않는다.

## 로드맵 목적

Time Tracker는 개인용 회사 적응 및 시간 기록 웹앱이다.

초기 목표는 사용자가 업무, 노트, 루틴, 회고, 목표, 미팅 기록을 안정적으로 관리할 수 있는 MVP를 완성하는 것이다.

중장기적으로는 시간 기록 데이터, 회고 데이터, 업무 기록 데이터를 기반으로 개인 생산성 분석, AI 회고 보조, 업무 패턴 인사이트, 모바일 앱 확장까지 발전시킨다.

## 제품 확장 방향

Time Tracker의 확장 방향은 아래 흐름을 따른다.

입력 → 기록 → 시간화 → 분석 → 회고 → 개선 → 자동화

각 단계의 의미는 다음과 같다.

- 입력: 사용자가 업무, 노트, 루틴, 회고, 목표, 미팅을 입력한다.
- 기록: 입력된 데이터를 사용자별로 안정적으로 저장한다.
- 시간화: 업무와 루틴에 실제 소요 시간을 연결한다.
- 분석: 누적된 기록을 기반으로 시간 사용 패턴을 보여준다.
- 회고: 하루 또는 한 주의 경험을 구조화해 정리한다.
- 개선: 다음 행동, 우선순위, 루틴 조정으로 연결한다.
- 자동화: AI와 알림을 통해 기록 부담을 줄이고 인사이트를 제공한다.

## 로드맵 단계 요약

| 단계 | 이름 | 핵심 목표 |
|---|---|---|
| Phase 1 | MVP 안정화 | 인증, CRUD, 기본 화면, 사용자별 데이터 저장 안정화 |
| Phase 2 | 기록 경험 고도화 | 노트, 업무, 루틴, 회고 입력 경험 개선 |
| Phase 3 | 시간 기록 강화 | 업무/루틴과 시간 기록 연결, 타이머 기능 도입 |
| Phase 4 | 분석 대시보드 | 일/주/월 단위 시간 사용 및 루틴 분석 |
| Phase 5 | AI 기능 확장 | 회고 요약, 인사이트 추천, 노트 자동 정리 |
| Phase 6 | 백엔드 확장 | FastAPI 기반 API 서버 및 비즈니스 로직 분리 |
| Phase 7 | 파일/이미지 저장 고도화 | Supabase Storage 또는 별도 파일 저장소 연동 |
| Phase 8 | 모바일 확장 | PWA, React Native, iOS/Android 앱 출시 검토 |
| Phase 9 | 외부 연동 | 캘린더, 알림, Export, 협업 도구 연동 |
| Phase 10 | 운영/배포 안정화 | QA, 모니터링, 오류 추적, 성능 최적화 |

## Phase 1. MVP 안정화

### 목표

사용자가 로그인한 뒤 주요 데이터를 생성, 조회, 수정, 삭제할 수 있는 기본 웹앱 구조를 안정화한다.

### 주요 작업

- Supabase Auth 기반 로그인
- 회원가입
- 로그아웃
- 비로그인 접근 제어
- 사용자별 데이터 접근
- 주요 도메인 CRUD
- query layer 구조 정리
- 기본 라우트 안정화
- Vercel 배포 가능 상태 유지

### 대상 도메인

- Auth
- Home Dashboard
- Notes
- Tasks
- Meetings
- Routines
- Goals
- Retrospectives
- Time Logs

### 완료 기준

- 로그인 사용자가 자신의 데이터만 볼 수 있다.
- 주요 도메인의 기본 CRUD가 동작한다.
- 컴포넌트에서 직접 Supabase table query를 호출하지 않는다.
- 타입 체크, 린트, 빌드가 통과한다.
- Vercel 배포가 가능하다.

## Phase 2. 기록 경험 고도화

### 목표

사용자가 기록을 더 쉽게 남기고, 중복 없이 정리할 수 있도록 입력 경험을 개선한다.

### 주요 작업

- 노트 카테고리 재정리
- 감정노트 회고 통합
- 상사노트, 동료노트 관계노트 통합
- 노트 이미지 첨부 UI
- 노트 검색, 태그, 정렬 개선
- 회고 포맷 정리
- 루틴 화면 캘린더 UI 추가
- 이번 주 루틴 현황 표시
- 빈 상태, 로딩 상태, 에러 상태 개선

### 노트 카테고리 방향

- 업무노트
- 관계노트
- 회사 용어사전
- 학습노트

### 회고 포맷 방향

- 작성일
- 오늘 배운 점
- 어려웠던 점
- 잘한 점
- 내일 적용할 행동
- 커뮤니케이션 인사이트
- 기술 인사이트
- 감정 관리 포인트

### 완료 기준

- 노트 기능의 중복 카테고리가 줄어든다.
- 회고와 감정 기록의 역할이 명확히 분리된다.
- 루틴 화면에서 월간/주간 흐름을 볼 수 있다.
- 모바일에서도 입력 화면이 깨지지 않는다.

## Phase 3. 시간 기록 강화

Time Tracker의 핵심 기능인 시간 기록을 강화한다.

사용자가 업무, 루틴, 미팅, 학습 활동에 실제 사용 시간을 연결할 수 있게 한다.

### 주요 작업

- 시간 기록 생성
- 시간 기록 수정
- 시간 기록 삭제
- 날짜별 시간 기록 조회
- 업무와 시간 기록 연결
- 루틴과 시간 기록 연결
- 카테고리별 시간 기록
- 수동 시간 입력
- 시작 시간, 종료 시간, 총 소요 시간 관리
- 타이머 시작, 일시정지, 종료 기능 검토

### 시간 기록 데이터 예시

- 기록 날짜
- 시작 시간
- 종료 시간
- 총 소요 시간
- 연결된 업무
- 연결된 루틴
- 카테고리
- 메모

### 완료 기준

- 사용자가 오늘 사용한 시간을 기록할 수 있다.
- 시간 기록이 업무 또는 루틴과 연결된다.
- 하루 총 기록 시간을 확인할 수 있다.
- 기존 업무, 루틴 CRUD가 깨지지 않는다.

## Phase 5. AI 기능 확장

### 목표

사용자의 기록 부담을 줄이고, 누적 데이터를 기반으로 개인화된 회고와 개선 인사이트를 제공한다.

### 주요 작업 후보

- 회고 자동 요약
- 주간 회고 요약
- 업무 시간 패턴 분석
- 감정 관리 포인트 추천
- 다음 행동 추천
- 노트 자동 태깅
- 업무 기록 요약
- 미팅 기록 요약
- 회의 액션아이템 추출
- 학습노트 핵심 키워드 추출
- 개인 생산성 코칭 메시지 생성

### AI 기능 우선순위

| 우선순위 | 기능 | 이유 |
|---|---|---|
| 1 | 회고 자동 요약 | 기존 회고 데이터와 가장 잘 연결됨 |
| 2 | 주간 인사이트 생성 | Time Tracker의 분석 가치 강화 |
| 3 | 노트 자동 태깅 | 기록 정리 부담 감소 |
| 4 | 미팅 액션아이템 추출 | 업무 생산성 기능 강화 |
| 5 | 개인 코칭 메시지 | 장기 차별화 기능 |

### AI 적용 원칙

- AI 기능은 사용자의 기록을 보조하는 역할로 제한한다.
- 사용자가 입력한 원문을 임의로 삭제하거나 변경하지 않는다.
- AI 결과는 사용자가 수정할 수 있어야 한다.
- AI 결과와 사용자 원본 기록은 구분한다.
- AI 기능은 명시된 마일스톤에 포함될 때만 구현한다.
- 비용, 응답 속도, 개인정보 처리를 고려한다.

### 완료 기준

- 회고 또는 노트 데이터를 기반으로 AI 요약을 생성할 수 있다.
- AI 결과를 사용자가 확인하고 수정할 수 있다.
- 기존 수동 기록 흐름이 깨지지 않는다.

## Phase 6. 백엔드 확장

### 목표

프로젝트 규모가 커질 경우 FastAPI 기반 별도 백엔드 서버를 도입해 비즈니스 로직, AI 처리, 파일 처리, 배치 작업을 분리한다.

### 현재 구조

초기 구조는 다음과 같다.

- Next.js App Router
- Supabase Auth
- Supabase PostgreSQL
- Supabase query layer
- Vercel 배포

### 확장 구조 후보

향후 구조는 다음과 같이 확장할 수 있다.

- Frontend: Next.js
- Backend API: FastAPI
- Auth: Supabase Auth
- Database: Supabase PostgreSQL
- Storage: Supabase Storage 또는 S3
- AI Service: OpenAI API 또는 별도 LLM API
- Deployment: Vercel + Render/Fly.io/AWS/GCP

### FastAPI 도입 후보 작업

- API 서버 분리
- 비즈니스 로직 분리
- AI 요약 API 구현
- 파일 업로드 API 구현
- 이미지 처리 API 구현
- OCR 처리 API 구현
- 배치 리포트 생성
- 주간 회고 자동 생성
- 알림 스케줄러
- 외부 연동 API 처리

### FastAPI 도입이 필요한 시점

아래 조건이 발생하면 FastAPI 도입을 검토한다.

- AI 기능이 많아져 서버 측 처리가 필요할 때
- 파일 업로드, 이미지 처리, OCR이 필요할 때
- 배치 작업 또는 스케줄러가 필요할 때
- Supabase query layer만으로 비즈니스 로직 관리가 복잡해질 때
- 외부 API 연동이 많아질 때
- 모바일 앱과 웹이 동일 API를 사용해야 할 때

### FastAPI 도입 전 확인할 것

- 현재 Next.js + Supabase 구조로 충분히 해결 가능한가
- API 서버 운영 비용을 감당할 수 있는가
- 인증 토큰 전달 구조가 명확한가
- RLS와 백엔드 권한 모델을 어떻게 조합할 것인가
- 배포 환경과 모니터링 체계가 준비되어 있는가

### 완료 기준

- 프론트엔드와 백엔드 API 책임이 분리된다.
- 기존 Supabase Auth 흐름과 충돌하지 않는다.
- 기존 사용자별 데이터 접근 원칙이 유지된다.
- AI, 파일, 배치 처리 로직을 백엔드에서 안정적으로 처리할 수 있다.

- 썸네일 표시
- 이미지 삭제
- 이미지 설명 입력
- 확대 미리보기

### 고도화 단계

- Supabase Storage 연동
- 이미지 영구 저장
- 이미지 URL DB 저장
- 이미지 삭제 시 Storage 파일 삭제
- 이미지 압축
- 이미지 확장자 검증
- 이미지 용량 제한
- 이미지 업로드 실패 처리
- 첨부파일 관리

### 추가 확장 후보

- 문서 이미지 OCR
- 이미지 내 텍스트 추출
- 회사 용어 자동 추출
- 에러 메시지 이미지 분석
- 학습 자료 이미지 요약

### 완료 기준

- 노트에 첨부한 이미지가 새로고침 후에도 유지된다.
- 사용자별 이미지 접근이 제한된다.
- 이미지 삭제 시 관련 데이터도 정리된다.
- 모바일에서도 이미지 첨부 경험이 안정적이다.

## Phase 8. 모바일 확장

### 목표

Time Tracker를 모바일에서도 자연스럽게 사용할 수 있도록 확장한다.

### 1단계: 모바일 웹 최적화

- 반응형 레이아웃
- 하단 탭 네비게이션
- 모바일 입력 폼 개선
- 터치 영역 최적화
- 모바일 이미지 첨부 UX 개선

### 2단계: PWA

- 홈 화면 설치
- 앱처럼 실행
- 기본 오프라인 화면
- 캐싱 전략 검토
- 웹 푸시 알림 검토

### 3단계: React Native 앱

- iOS 앱
- Android 앱
- 공통 API 서버 사용
- 앱 전용 푸시 알림
- 앱스토어/플레이스토어 출시 검토

### 모바일 앱 도입 전 확인할 것

- 웹앱에서 핵심 사용성이 검증되었는가
- 매일 사용할 만큼 기록 가치가 있는가
- 사용자 인증과 데이터 동기화 구조가 안정적인가
- API 서버가 필요한 수준까지 기능이 확장되었는가
- 푸시 알림, 오프라인 저장 등 앱 고유 기능이 필요한가

### 완료 기준

- 모바일 웹에서 핵심 기능을 불편 없이 사용할 수 있다.
- PWA 또는 앱 확장 여부를 판단할 수 있는 사용성 기준이 확보된다.

## 구현 보류 기준

아래 조건에 해당하면 구현을 보류한다.

- 현재 마일스톤에 포함되지 않은 기능
- 기존 기능 안정성을 해칠 가능성이 큰 기능
- DB schema 변경이 필요한데 명시 요청이 없는 기능
- RLS policy 변경이 필요한데 명시 요청이 없는 기능
- 인증 구조 변경이 필요한 기능
- 불필요한 라이브러리 추가가 필요한 기능
- 현재 MVP 가치와 직접 연결되지 않는 기능
- 개발 범위가 너무 큰 기능

## 문서 운영 원칙

이 문서는 중장기 방향을 설명한다.

실제 구현은 아래 문서 우선순위를 따른다.

1. 사용자 작업 요청
2. `AGENTS.md`
3. `docs/MILESTONES.md`
4. 관련 `agents/*-agent.md`
5. `docs/HARNESS.md`
6. `docs/PRD.md`
7. `docs/ROADMAP.md`

문서 간 충돌이 있을 경우 현재 작업 범위를 가장 우선한다.

## AGENTS.md와의 관계

`AGENTS.md`는 작업자가 반드시 지켜야 할 프로젝트 규칙 문서다.

`ROADMAP.md`는 향후 확장 가능성을 설명하는 참고 문서다.

따라서 `ROADMAP.md`에 포함된 기능이라도 `AGENTS.md`, `docs/MILESTONES.md`, 사용자 요청 범위에 포함되지 않으면 구현하지 않는다.

## PRD.md와의 관계

`docs/PRD.md`는 제품 요구사항과 기능 방향을 설명한다.

`ROADMAP.md`는 PRD 이후의 확장 흐름과 장기 계획을 설명한다.

PRD와 ROADMAP에 포함된 기능은 모두 참고 사항이며, 실제 구현은 마일스톤 단위로만 진행한다.

## MILESTONES.md와의 관계

`docs/MILESTONES.md`는 실제 구현 순서를 정의한다.

따라서 작업자는 ROADMAP보다 MILESTONES를 우선한다.

ROADMAP에 있는 기능은 다음 마일스톤 설계 시 참고할 수 있지만, 현재 마일스톤에 포함되지 않으면 구현하지 않는다.
