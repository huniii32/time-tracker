# HARNESS

## 하네스 목적

하네스는 Codex가 작은 범위로 구현하고 검증하도록 강제하는 작업 기준이다. 기능을 많이 만드는 것보다, 현재 마일스톤을 정확히 끝내고 검증하는 것을 우선한다.

## 작업 흐름

1. 작업 범위 확인
2. 해당 마일스톤 확인
3. 관련 에이전트 문서 확인
4. 구현
5. 타입 체크
6. 린트
7. 빌드
8. 기능별 수동 검증
9. 범위 초과 여부 확인
10. 완료 리포트 작성

## 작업 시작 전 확인사항

- `AGENTS.md`를 읽는다.
- `docs/MILESTONES.md`에서 이번 마일스톤을 확인한다.
- 관련 `agents/*-agent.md`를 확인한다.
- `docs/PRD.md`는 참고 문서로만 사용한다.
- `package.json`이 있는지 확인한다.
- 기존 파일을 임의로 삭제하지 않는다.

## 작업 중 금지사항

- 현재 마일스톤 밖 기능 구현
- 요청하지 않은 CRUD 구현
- 요청하지 않은 인증 구현
- 요청하지 않은 Supabase 연결
- 불필요한 라이브러리 추가
- 대규모 리팩토링
- 검증 없이 완료 보고

## 현재 검증 명령어 상태

현재 이 프로젝트에는 `package.json`이 없다. 따라서 실제 명령어는 TODO로 남긴다.

향후 `package.json` 생성 후 아래 항목을 확인한다.

```text
TODO: typecheck command
TODO: lint command
TODO: build command
TODO: test command, if needed
```

## 작업 완료 후 검증 기준

- 타입 체크가 통과해야 한다.
- 린트가 통과해야 한다.
- 빌드가 통과해야 한다.
- 주요 화면이 렌더링되어야 한다.
- 로그인 상태별 접근 흐름을 확인해야 한다.
- 모바일 화면 레이아웃을 확인해야 한다.
- 작업 범위 외 기능을 구현하지 않았는지 확인해야 한다.

## 타입 체크 기준

- TypeScript 에러를 남기지 않는다.
- 임시 `any` 사용을 피한다.
- 타입 정의는 `src/types` 또는 기능별 타입 파일로 분리한다.

## 린트 기준

- 프로젝트에 정의된 린트 명령을 사용한다.
- 린트 실패 시 완료 보고하지 않는다.
- 린트 규칙을 우회하기 위해 불필요한 disable 주석을 추가하지 않는다.

## 빌드 기준

- 배포 가능한 빌드가 통과해야 한다.
- 환경 변수 누락이 있으면 문서화하고 완료 보고에 포함한다.
- 빌드 실패를 남긴 채 다음 작업으로 넘기지 않는다.

## 인증 검증 기준

- 비로그인 사용자는 보호 화면에 접근할 수 없어야 한다.
- 로그인 성공 시 홈으로 이동해야 한다.
- 로그아웃 시 로그인 화면으로 이동해야 한다.
- 세션 새로고침 동작을 확인해야 한다.

## CRUD 검증 기준

- 생성 후 목록에 보인다.
- 새로고침 후 데이터가 유지된다.
- 수정 후 변경값이 반영된다.
- 삭제 후 목록에서 사라진다.
- 현재 로그인한 사용자 본인 데이터만 접근한다.

## UI 검증 기준

- 모바일 우선 레이아웃을 유지한다.
- 하단 탭이 주요 조작을 가리지 않는다.
- 입력 폼 텍스트가 잘리지 않는다.
- 카드 안의 텍스트와 버튼이 겹치지 않는다.
- Empty State를 제공한다.

## 범위 초과 검증 기준

작업 완료 전 아래를 확인한다.

- 이번 마일스톤 밖 기능을 구현하지 않았는가?
- 요청하지 않은 파일을 수정하지 않았는가?
- PRD 전체를 한 번에 구현하려고 하지 않았는가?
- 불필요한 라이브러리를 추가하지 않았는가?

## 완료 리포트 형식

```markdown
### 구현한 것

### 생성/수정한 파일

### 구현하지 않은 것

### 검증 결과

### 현재 프로젝트 상태

### 다음 추천 작업

### 주의해야 할 점
```
