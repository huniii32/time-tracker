# MILESTONES.md

# Time Tracker 기능 구현 순서

> 이 문서는 Time Tracker의 실제 구현 순서를 정의한다.
> PRD와 ROADMAP에 포함된 기능이라도 현재 마일스톤에 포함되지 않으면 구현하지 않는다.
> 한 번의 작업은 하나의 마일스톤 또는 사용자가 명시한 작업 범위만 진행한다.

## 프로젝트 개요

Time Tracker는 개인용 회사 적응 및 시간 기록 웹앱이다.

사용자는 로그인 후 업무, 노트, 루틴, 회고, 목표, 미팅 기록을 단계적으로 관리하고, 자신의 시간 사용 패턴과 회사 적응 과정을 구조적으로 기록할 수 있다.

핵심 흐름은 아래와 같다.

입력 → 기록 → 시간화 → 회고 → 개선

## 마일스톤 원칙

- 전체 기능을 한 번에 구현하지 않는다.
- 각 단계는 하나의 마일스톤으로만 진행한다.
- 이전 마일스톤이 검증을 통과해야 다음 마일스톤으로 넘어간다.
- 현재 마일스톤에 포함되지 않은 기능은 구현하지 않는다.
- PRD는 제품 방향 참고 문서이며, 실제 구현 범위는 이 문서를 따른다.
- ROADMAP은 중장기 확장 참고 문서이며, 현재 구현 범위로 해석하지 않는다.
- 구현 후 반드시 `docs/HARNESS.md` 기준으로 검증한다.
- 검증 실패 시 완료로 보고하지 않는다.
- 작업 완료 후 반드시 `logs/` 폴더에 작업 로그를 남긴다.

## 공통 유지 원칙

모든 마일스톤에서 아래 원칙을 유지한다.

- 기존 인증 흐름 유지
- 기존 라우트 구조 유지
- 기존 Supabase query layer 구조 유지
- 현재 로그인한 사용자 `user.id` 기준 데이터 접근
- DB row의 `user_id` 기준 사용자 데이터 분리
- 컴포넌트 내 직접 Supabase table query 금지
- `src/app`, `src/components` 내 직접 `.from(...)` 호출 금지
- 타입 안정성 유지
- 모바일 반응형 유지
- Time Tracker의 네이비/화이트 UI 방향 유지
- 불필요한 라이브러리 추가 금지
- DB schema 변경은 해당 마일스톤에 명시된 경우에만 진행
- RLS policy 변경은 해당 마일스톤에 명시된 경우에만 진행

## 현재 제외되는 장기 확장 기능

아래 기능은 ROADMAP에 포함될 수 있지만, 이 문서의 해당 마일스톤에 명시되기 전까지 구현하지 않는다.

- FastAPI 백엔드 전환

- React Native 앱 구현

- Supabase Storage 연동

# Milestone 2: 데이터 모델과 Query Layer

## 목표

기능 구현 전에 사용자별 데이터 접근 구조를 안정화한다.

각 도메인 데이터가 `user_id` 기준으로 분리되고, 컴포넌트가 직접 Supabase table query를 호출하지 않도록 query layer 구조를 만든다.

## 포함 범위

- profiles 데이터 구조
- notes 데이터 구조
- tasks 데이터 구조
- meetings 데이터 구조
- routines 데이터 구조
- goals 데이터 구조
- retrospectives 데이터 구조
- time_logs 데이터 구조
- 기능별 TypeScript 타입 정의
- 기능별 query layer 파일 생성
- 사용자별 `user_id` 기반 접근 구조
- 기본 CRUD query 함수 skeleton

## 제외 범위

- 전체 화면 UI 구현
- 실제 도메인별 CRUD 화면 구현
- 복잡한 대시보드 계산
- 차트
- AI 기능
- 외부 API 연동
- Supabase Storage 연동
- 파일 업로드 기능

## 완료 기준

- 주요 데이터 타입이 정의되어 있다.
- 기능별 query layer 파일이 준비되어 있다.
- 모든 주요 데이터 구조가 `user_id` 기준 접근을 고려한다.
- 컴포넌트에서 직접 `.from(...)`을 호출하지 않는 구조가 준비되어 있다.
- 타입 체크가 통과한다.
- 린트가 통과한다.
- Codex 환경에서는 build를 실행하지 않는다. build 검증은 사용자가 로컬 또는 CI에서 별도로 수행한다.

- 학습노트 → 학습노트

# Milestone 9: 기록 경험 고도화

## 목표

노트 작성 시 이미지를 첨부할 수 있게 하고, 루틴 화면에는 월간 캘린더와 이번 주 루틴 현황을 추가한다.

## 포함 범위

- 노트 작성/수정 화면 이미지 첨부 UI
- 다중 이미지 첨부
- 이미지 썸네일 미리보기
- 이미지 삭제
- 이미지 확대 미리보기
- 이미지별 설명 입력
- 루틴 화면 월간 캘린더 UI
- 오늘 날짜 강조
- 날짜별 루틴 달성 여부 표시
- 이번 주 루틴 현황 보드
- 월~일 기준 루틴 완료 여부 표시
- 데일리 시간 탭 내부 구조 개선
- 빈 상태 UI 개선

## 데일리 시간 탭 내부 구조

1. 월간 캘린더
2. 이번 주 루틴 현황
3. 오늘의 시간 기록

## 이미지 첨부 초기 구현 원칙

- 백엔드 저장 없이 프론트엔드 미리보기 중심으로 구현할 수 있다.
- Supabase Storage 연동은 하지 않는다.
- 파일 업로드 서버 API는 만들지 않는다.
- 추후 서버 연동이 가능하도록 이미지 관련 로직은 함수로 분리한다.

## 제외 범위

- Supabase Storage 연동
- 파일 업로드 서버 API
- 이미지 영구 저장
- OCR
- AI 이미지 분석
- 외부 캘린더 연동
- Google Calendar 연동
- Apple Calendar 연동
- 푸시 알림
- 홈 대시보드 고도화
- 차트 추가
- DB schema 변경
- RLS policy 변경
- 인증 구조 변경
- 대규모 UI 리디자인

## 완료 기준

- 노트 작성 시 이미지를 첨부할 수 있다.
- 첨부 이미지를 썸네일로 볼 수 있다.
- 첨부 이미지를 삭제할 수 있다.
- 첨부 이미지를 확대 미리보기로 볼 수 있다.
- 이미지별 설명을 입력할 수 있다.
- 루틴 화면 상단에 월간 캘린더가 표시된다.
- 캘린더에서 오늘 날짜가 강조된다.
- 날짜별 루틴 달성 여부가 표시된다.
- 캘린더 아래에 이번 주 루틴 현황이 표시된다.
- 기존 데일리 시간 기록 영역이 유지된다.
- 기존 CRUD 기능이 깨지지 않는다.
- 타입 체크가 통과한다.
- 린트가 통과한다.
- Codex 환경에서는 build를 실행하지 않는다. build 검증은 사용자가 로컬 또는 CI에서 별도로 수행한다.
