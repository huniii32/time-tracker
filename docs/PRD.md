# PRD: Time Tracker / 회사 적응 기록 웹앱

> 이 문서는 제품 요구사항 참고 문서다.
> 이 문서를 한 번에 모두 구현하지 않는다.
> 실제 구현 범위는 항상 `docs/MILESTONES.md` 기준으로 단계별 진행한다.
> PRD에 포함된 기능이라도 현재 마일스톤에 포함되지 않으면 구현하지 않는다.

## 제품 개요

Time Tracker는 개인용 회사 적응 및 시간 기록 웹앱이다.

사용자는 로그인 후 업무, 노트, 루틴, 회고, 목표, 미팅 기록을 단계적으로 관리하고, 자신의 시간 사용 패턴과 회사 적응 과정을 구조적으로 기록할 수 있다.

이 서비스는 단순한 업무 메모 앱이 아니라, 사용자가 회사 생활에서 발생하는 정보와 경험을 기록하고, 시간을 기준으로 정리하며, 회고를 통해 다음 행동을 개선하는 것을 목표로 한다.

핵심 흐름은 다음과 같다.

입력 → 기록 → 시간화 → 회고 → 개선

## 제품 목적

Time Tracker는 아래 문제를 해결한다.

- 회사 적응 과정에서 업무, 사람, 용어, 회의, 학습 내용이 흩어지는 문제
- 하루 동안 무엇을 했는지 기록은 하지만 실제 시간을 어떻게 썼는지 파악하기 어려운 문제
- 감정 기록, 학습 기록, 업무 기록, 회고가 서로 분리되어 중복되는 문제
- 루틴과 목표를 관리하지만 달성 흐름을 한눈에 보기 어려운 문제
- 회고가 단순 감상으로 끝나고 다음 행동 개선으로 이어지지 않는 문제

## 사용자

주요 사용자는 개인 사용자 1인이다.

대상 사용자는 다음과 같다.

- 회사 적응 과정을 체계적으로 기록하려는 신입 구성원
- 이직 또는 직무 전환 초기에 업무 흐름을 정리하고 싶은 사용자
- 회사 생활에서 업무, 관계, 루틴, 회고를 함께 관리하고 싶은 사용자
- 하루 시간 사용과 생산성 패턴을 기록하고 개선하고 싶은 사용자

## 화면 구조

하단 탭은 5개로 구성한다.

1. 홈
2. 노트
3. 업무
4. 루틴
5. 회고

각 탭은 다음 역할을 가진다.

### 홈

홈은 사용자가 오늘 해야 할 일과 최근 기록을 빠르게 확인하는 시작 화면이다.

주요 정보는 다음과 같다.

- 입사 D+일차
- 오늘의 체크리스트
- 오늘의 업무
- 오늘의 루틴
- 최근 기록
- 주간 목표 진행률
- 오늘의 시간 기록 요약
- 회고 작성 여부

### 노트

노트는 회사 생활에서 발생하는 정보를 목적별로 기록하는 공간이다.

노트 카테고리는 중복을 줄이고 아래 4개로 정리한다.

- 업무노트
- 관계노트
- 회사 용어사전
- 학습노트

기존 카테고리는 아래 기준으로 통합한다.

- 회사노트 → 업무노트
- 상사노트 → 관계노트
- 동료노트 → 관계노트
- 회사 단어 사전 → 회사 용어사전
- 학습노트 → 학습노트
- 감정노트 → 회고의 감정 관리 포인트로 통합

노트 주요 기능은 다음과 같다.

- 노트 목록 조회
- 노트 생성
- 노트 상세 조회
- 노트 수정
- 노트 삭제
- 카테고리 선택
- 태그 입력
- 날짜 정렬
- 검색
- 이미지 첨부
- 이미지 미리보기
- 이미지 삭제
- 이미지 설명 입력

### 업무

업무는 사용자가 해야 할 일과 진행 상태를 관리하는 공간이다.

주요 기능은 다음과 같다.

- 업무 목록 조회
- 업무 생성
- 업무 상세 조회
- 업무 수정
- 업무 삭제
- 상태 관리
- 우선순위 관리
- 마감기한 관리
- 업무 설명 기록
- 업무와 시간 기록 연결

상태값 예시는 다음과 같다.

- 예정
- 진행중
- 완료
- 보류

우선순위 예시는 다음과 같다.

- 낮음
- 보통
- 높음
- 긴급

### 미팅

미팅은 회의 내용, 결정사항, 액션아이템을 기록하는 공간이다.

미팅은 업무 도메인과 연결될 수 있지만, 실제 구현 범위는 마일스톤에 따른다.

주요 기능은 다음과 같다.

- 미팅 목록 조회
- 미팅 생성
- 미팅 상세 조회
- 미팅 수정
- 미팅 삭제
- 참석자 기록
- 회의 목적 기록
- 주요 논의 내용 기록
- 결정사항 기록
- 액션아이템 기록
- 후속 업무 연결

### 루틴

루틴은 사용자의 반복 행동과 시간 사용을 관리하는 공간이다.

루틴 화면은 아래 탭 구조를 가진다.

- 데일리 시간
- 위클리 피드백
- 목표 통합

데일리 시간 탭은 아래 순서로 구성한다.

1. 월간 캘린더
2. 이번 주 루틴 현황
3. 오늘의 시간 기록

루틴 주요 기능은 다음과 같다.

- 루틴 목록 조회
- 루틴 생성
- 루틴 수정
- 루틴 삭제
- 루틴 완료 체크
- 월간 캘린더 표시
- 날짜별 루틴 달성 여부 표시
- 이번 주 루틴 현황 표시
- 오늘의 시간 기록 표시
- 루틴별 소요 시간 기록

월간 캘린더는 외부 캘린더 연동이 아니라 내부 표시용 캘린더다.

Google Calendar, Apple Calendar 등 외부 캘린더 연동은 제외한다.

### 회고

회고는 하루 동안의 경험을 정리하고 다음 행동으로 연결하는 공간이다.

감정노트는 별도 기능으로 두지 않고 회고의 `감정 관리 포인트`로 통합한다.

회고 기본 포맷은 다음과 같다.

- 작성일
- 오늘 배운 점
- 어려웠던 점
- 잘한 점
- 내일 적용할 행동
- 커뮤니케이션 인사이트
- 기술 인사이트
- 감정 관리 포인트

Time Tracker 목적에 맞게 추후 아래 항목을 추가할 수 있다.

- 오늘의 주요 시간 사용
- 예상보다 오래 걸린 일
- 가장 집중했던 시간
- 내일 줄이고 싶은 시간 사용
- 내일 늘리고 싶은 시간 사용

단, 회고 포맷 확장은 해당 마일스톤에 포함될 때만 구현한다.

## 이미지 첨부 요구사항

노트 작성 및 수정 화면에서는 이미지를 첨부할 수 있다.

이미지 첨부 기능은 다음 요구사항을 가진다.

- 한 노트에 여러 장의 이미지를 첨부할 수 있다.
- 업로드 가능한 파일 형식은 jpg, jpeg, png, webp로 제한한다.
- 이미지 선택 후 썸네일 미리보기를 제공한다.
- 첨부 이미지를 삭제할 수 있다.
- 첨부 이미지를 클릭하면 확대 미리보기를 제공한다.
- 이미지마다 선택적으로 설명을 입력할 수 있다.
- 모바일 화면에서도 깨지지 않게 표시한다.

이미지 첨부 기능은 단계적으로 구현한다.

초기 구현에서는 프론트엔드 미리보기 중심으로 구현할 수 있다.

실제 파일 저장소 연동, Supabase Storage 연동, 서버 업로드 API 구현은 별도 마일스톤에서 진행한다.

인증 기능은 Supabase Auth를 사용한다.

요구사항은 다음과 같다.

- 이메일/비밀번호 로그인
- 회원가입
- 로그아웃
- 비로그인 접근 제어
- 현재 로그인한 사용자 정보 조회
- 사용자 프로필 관리

OAuth 소셜 로그인은 제외한다.

## 데이터 저장 원칙

모든 주요 데이터는 Supabase PostgreSQL에 저장한다.

모든 주요 데이터는 아래 공통 필드를 가진다.

- id
- user_id
- created_at
- updated_at

모든 데이터 접근은 현재 로그인한 사용자의 `user_id` 기준으로 제한한다.

Supabase RLS로 사용자 데이터 격리를 보장한다.

데이터 접근 로직은 UI 컴포넌트와 분리한다.

컴포넌트에서 Supabase table query를 직접 호출하지 않는다.

`src/app`, `src/components` 내부에서 직접 `.from(...)`을 호출하지 않는다.

데이터 조회, 생성, 수정, 삭제는 query layer 또는 domain query 함수를 통해 수행한다.

## 마일스톤 구현 원칙

이 PRD는 제품 전체 방향을 설명한다.

실제 구현은 `docs/MILESTONES.md` 기준으로 진행한다.

작업자는 PRD 전체를 한 번에 구현하지 않는다.

현재 마일스톤에 포함되지 않은 기능은 구현하지 않는다.

모호한 기능은 구현 범위를 넓히지 않고 최소 범위로 제한한다.

## 검증 원칙

각 마일스톤 작업 후에는 반드시 `docs/HARNESS.md` 기준으로 검증한다.

검증 항목은 다음을 포함할 수 있다.

- 타입 체크
- 린트
- 빌드
- 주요 라우트 렌더링
- 현재 작업 대상 기능 동작
- 기존 CRUD 기능 유지 여부
- 로그인 사용자 기준 데이터 접근 여부
- query layer 사용 여부
- 컴포넌트 내 직접 Supabase 호출 여부
- 범위 밖 기능 구현 여부
- 불필요한 라이브러리 추가 여부

검증 실패 시 완료로 보고하지 않는다.

## PRD 원문 위치

사용자가 작성한 상세 PRD는 이후 이 문서에 추가할 수 있다.

추가하더라도 전체를 한 번에 구현하지 않고 `docs/MILESTONES.md` 기준으로 나누어 진행한다.

## 추후 확장 후보

아래 기능들은 제품 방향상 고려할 수 있지만, 현재 PRD 기준 즉시 구현 대상은 아니다.

실제 구현 여부와 순서는 `docs/MILESTONES.md` 또는 별도 `docs/ROADMAP.md`에서 결정한다.
